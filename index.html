01 · 项目概览01 · Overview

研究背景与问题定义Background and problem statement

现代网络经常同时包含有线以太网、Wi-Fi、蜂窝接入、边缘节点和云端服务。不同链路在带宽、传播时延、队列长度、丢包率和稳定性方面差异明显，而业务流量又可能在很短时间内发生变化。传统链路状态路由协议强调全网一致性和稳定收敛，但在高频波动、边缘计算或交互式业务场景中，固定周期更新与全局信息传播可能带来额外控制开销。Modern networks often combine wired Ethernet, Wi-Fi, cellular access, edge nodes, and cloud services. Links differ markedly in bandwidth, propagation delay, queue length, loss rate, and stability, while traffic can shift within very short intervals. Traditional link-state protocols emphasize network-wide consistency and stable convergence, but under high-frequency fluctuation, edge computing, or interactive workloads, fixed-period updates and global information flooding can incur extra control overhead.

CaelumRoute 的核心问题不是宣称替代 OSPF、BGP 或 ECMP，而是探索一种补充性的控制平面机制：能否通过有限、可解释的拓扑采样与链路遥测摘要，识别真正影响路径质量的变化，并只对受影响的路径执行增量重计算。研究同时关注性能收益、控制开销、路径稳定性、公平性和部署复杂度，避免只优化单一延迟指标。CaelumRoute does not claim to replace OSPF, BGP, or ECMP. It explores a complementary control-plane mechanism: whether a limited, interpretable set of topology samples and telemetry summaries can identify the changes that actually affect path quality, and recompute only the affected paths. The study jointly considers performance gains, control overhead, path stability, fairness, and deployment complexity — rather than optimizing a single latency metric.

1.1 研究对象1.1 Scope of study

研究对象包括异构链路、动态节点负载、突发流量、随机链路失效和多路径转发。控制平面接收的观测信息可以来自主动探测、设备计数器、队列遥测或仿真环境中的链路状态。项目不假设所有设备都支持同一种协议，而是把可编程交换机、传统路由器和软件交换节点视为不同能力等级。The study covers heterogeneous links, dynamic node load, bursty traffic, random link failures, and multi-path forwarding. Observations reaching the control plane may come from active probing, device counters, queue telemetry, or link state in simulation. The project does not assume every device supports the same protocol; programmable switches, legacy routers, and software switches are treated as distinct capability tiers.

1.2 研究边界1.2 Boundaries

当前阶段定义研究问题、设计框架和评估方法，不声称已经形成可投入生产的网络协议。安全性、跨自治系统策略、超大规模互联网部署以及硬件数据平面实现不属于第一阶段验证范围，但会作为研究限制明确记录。The current stage defines the research questions, design framework, and evaluation methods; it does not claim a production-ready protocol. Security, inter-AS policy, hyperscale Internet deployment, and hardware data-plane implementation are outside the first-stage scope, and are recorded explicitly as limitations.

02 · 系统设计02 · System design

四层研究架构A four-layer architecture

2.1

观测层 · Network ObservationObservation layer

收集链路时延、丢包率、可用带宽、队列长度、节点负载和故障事件，并记录采样频率、测量误差和时间戳，以便后续判断观测值是否仍然有效。Collects link delay, loss rate, available bandwidth, queue length, node load, and failure events — recording sampling frequency, measurement error, and timestamps so validity can be judged later.

2.2

采样层 · CATSSampling layer · CATS

对原始拓扑状态进行分层采样与摘要聚合，重点研究采样节点选择、变化阈值、摘要过期时间和局部信息传播范围。Performs hierarchical sampling and summary aggregation over raw topology state, focusing on sampling-node selection, change thresholds, summary expiry, and the scope of local information propagation.

2.3

决策层 · Incremental DecisionDecision layer

根据受影响范围重新计算候选路径，在时延、稳定性、控制开销和负载均衡之间进行多目标权衡，并保留触发原因和评分依据。Recomputes candidate paths for the affected scope, trading off delay, stability, control overhead, and load balancing — while retaining the trigger cause and scoring rationale.

2.4

执行层 · Route EnforcementEnforcement layer

将路径更新映射到仿真节点、软件交换机或支持 OpenFlow/P4 的实验环境。第一阶段优先验证算法行为。Maps path updates onto simulated nodes, software switches, or OpenFlow/P4 environments. The first stage prioritizes validating algorithm behavior.

候选工作流程Candidate workflow

  1. 采集：Collect: 按固定周期与事件触发方式采集链路和节点状态。gather link and node state on a fixed period and on events.
  2. 清洗：Clean: 处理异常值、时钟偏差、缺失值和短时抖动。handle outliers, clock skew, missing values, and short-lived jitter.
  3. 摘要：Summarize: 将局部观测压缩为带时间戳和置信度的状态摘要。compress local observations into timestamped, confidence-tagged summaries.
  4. 识别：Identify: 定位可能影响服务级目标的拓扑变化。locate topology changes that may affect service-level objectives.
  5. 计算：Compute: 只为受影响流或目的前缀生成候选路径。generate candidate paths only for affected flows or prefixes.
  6. 约束：Constrain: 应用环路避免、路径切换抑制和容量约束。apply loop avoidance, switch damping, and capacity limits.
  7. 执行：Enforce: 分阶段应用路由变更并记录控制平面事件。apply route changes in stages and log control-plane events.
  8. 评估：Evaluate: 比较端到端指标、稳定性、开销和失败案例。compare end-to-end metrics, stability, overhead, and failure cases.

03 · 研究问题03 · Research questions

待验证的问题与假设Questions and hypotheses to test

RQ1：采样是否能降低控制开销？RQ1: Does sampling reduce control overhead?

比较全量链路状态传播与分层采样摘要在不同拓扑规模、采样周期和波动强度下的消息数量、字节数与控制平面 CPU 时间。Compare full link-state flooding vs. hierarchical sampled summaries in message count, bytes, and control-plane CPU time across topology sizes, sampling periods, and fluctuation intensities.

RQ2：局部更新是否会损害路径质量？RQ2: Does local updating hurt path quality?

评估增量更新相对全局最短路径在端到端时延、跳数、丢包率、链路利用率和路径伸长率方面的差异。Assess incremental updates vs. global shortest paths in end-to-end delay, hop count, loss rate, link utilization, and path stretch.

RQ3：如何避免频繁切换？RQ3: How to avoid frequent switching?

研究滞回阈值、最短保持时间、变化置信度和批量更新窗口对路径抖动、乱序和短流完成时间的影响。Study how hysteresis thresholds, minimum hold time, change confidence, and batching windows affect path flapping, reordering, and short-flow completion time.

RQ4：异构能力如何影响部署？RQ4: How does heterogeneity affect deployment?

区分仅支持传统路由的节点、软件交换节点和可编程数据平面节点，分析不同部署比例下可以实现的功能与额外复杂度。Separate legacy-routing nodes, software switches, and programmable data-plane nodes, and analyze what is achievable — and at what added complexity — under different deployment ratios.

06 · 常见问题 FAQ06 · FAQ

CaelumRoute 研究的核心问题是什么？What is the core research question?

在异构、低时延、高波动的网络中，能否通过有限且可解释的拓扑采样与遥测摘要，识别真正影响路径质量的变化，并只对受影响的路径进行增量重计算，从而在性能、稳定性与控制开销之间取得更好的平衡。In heterogeneous, low-latency, high-fluctuation networks, can a limited and interpretable set of topology samples and telemetry summaries identify the changes that actually affect path quality — and recompute only the affected paths — to better balance performance, stability, and control overhead?

CaelumRoute 与 OSPF、ECMP 的关系是什么？How does it relate to OSPF and ECMP?

OSPF 和 ECMP 作为评估基线。CaelumRoute 研究在链路质量波动和异构接入条件下，利用遥测摘要与增量路径计算减少不必要的全局更新，其效果通过公开的 ns-3 实验进行评估。OSPF and ECMP serve as evaluation baselines. CaelumRoute studies whether telemetry summaries and incremental path computation can reduce unnecessary global updates under link-quality fluctuation and heterogeneous access, with effects assessed through open ns-3 experiments.

CATS 是什么？What is CATS?

CATS 是 Caelum-Aware Topology Sampling 的名称，描述对拓扑状态、链路质量和节点负载进行分层采样与摘要聚合的机制，是 CaelumRoute 采样层的核心设计。CATS stands for Caelum-Aware Topology Sampling — the mechanism for hierarchical sampling and summary aggregation over topology state, link quality, and node load. It is the core of CaelumRoute's sampling layer.
